# Einführung
> Abstraktionen, Abstraktionsschichten
- Komplexe Systeme können durch ==Abstreaktionsschichten== besser beschrieben werden
	- Für eine Schicht irrelevante Funktionalität wird abstrahiert
	- Abstraktionen erlauben dem Betrachter sich auf relevante Bereiche zu konzentrieren
- Abstreaktionsschichten durch ==Schnittstellen== (Interfaces) voneinander getrennt
	- Details der Realisierung der unteren Schichten dadurch verborgen (Black Box)
<br/>

> Beispiel: Systemcalls
- Syscalls können als Abstreaktion ihrer Low-Level-Implementierung innerhalb des BS-Kerns angesehen werden
- Implementierungsdetails durch das ==System Call Interface== verborgen 
<br/>

> ==Hardware-Details== können durch ==Interfaces== abstrahiert werden
- Das Interface definiert die Software-Sich auf die Hardware
- Siehe (Hierarchien von) Treiber(n)
<br/>

-> Hardware-Funktionalität kann duch SW ==simuliert/emuliert== werden
<br/>

> Die Sicht auf ein HW-System wird durch unterschiedliche Schnittstellen definiert
- ==Instruction Set Architecture== (ISA): Interface zwischen dem BS und Hardware
	- Die ISA besteht aus User- und System-ISA
- ==Application Binary Interface== (ABI): Interface zwischen Anwendungen und dem Betriebssystem
	- Das ABI besteht aus dem Sys Call Interface und der User ISA
- -> Die Siche auf das System hängt von der SW-Perspektive ab
<br/>

> Virtuelle Laufzeitumgebung
- Ein HW-Sys (physische Machine) biete eine Laufzeitumgebung für die darüberliegende Software
	- BS virtualisiert bereits: Prozesse, virtueller Speicher und Abstraktionen
- Eine Virtuelle Maschine implementiert eine ==virtuelle Laufzeitumgebung==
	- Abbilden von virtuellen auf physische Ressourcen
		- CPU-Zustand, HWgeräte, etc.
		- Emulieren der virtuellen ABI bzw. ISA
<br/>

> Abhängig davon, welche Sicht auf das System die Lautzeitumgebung implementiert, unterscheidet man
- System Virtual Machines
- Process Virtual Machines

# Anforderungen an Virtualisierung
> Equivalence / Fidelity  
- BS und Anwendungen funktionieren ohne Änderungen
	- Ggf. minimale Änderungen/Ergänzungen am Gast-BS
- Verhalten sich genauso wie auf einem nativen System
<br/>

> Resource Control / Safety (Isolation)
- Hypervisor muss in Kontrolle sein 
- Hypervisor schützt Ressourcen vor und Gast-BS gegeneinander
<br/>

> Efficienty / Performance 
- Die meisten Instruktionen müssen ohne Intervention des Virtualisierungssys ausgeführt werden
- Virtualisierungss fügt eine weiter Software-Schicht hinzu (Overhead)
	- Wie auch ein BS versucht das VS den Overhead zu minimieren
	- Bsp für VMware (nach Alex Snoeren, CSE 120, Lecture 17)
		- CPU-intensive Anwendungen: 2-10% Overhead
		- I/O-intensive Anwendungen: 25-60% Overhead


# OS-Level-Virtualisation: Container
- Leichtgewichtige Virtualisierungsmöglichkeit
- Container stellen eine Laufzeitumgebung für User-Space-Prozesse bereit
	- Prinzipiell beliebig viele Prozesse
- Verwenden Services des BS-Kerns
	- Container verwenden somit dieselbe ISA wie der Host
- Der BS-Kern wird zwischen dem Host und Gast ==geteilt== 
	- D.h. BS wird ==nicht virtualisiert==
	- Bsp. Solaris Zones, FreeBSD Jails, Lunix Containers (LXC), Docker, rkt
<br/>

> Problem: Da alle Container den ==gleichen BS-Kern== teilen, gibt es keine HWvirtualisierung, die Prozesse strikt voneinander trennt 
- -> Isolation von Prozessen
	- Kontrolle einschränken der verfügbaren Ressourcen
	- Dem Container die Illusion geben, die Resourcen exklusiv zu nutzen (dass kein andere sie mitbenutzt)

### Linux Namespaces
- Namensräume zur Benennung und Adressierung von Ressourcen 
<br/>

- In traditionellen (Unix-basierten) Systemen werden einige Ressourcen global verwaltet
	- Eindeutige PIDs pro Prozess
	- Eindeutige User IDs
	- Host- und NIS-Domainnamen, IP-Adressen, ...
- Werdem vom Kernel in einer globalen Liste verwaltet
	- User Prozesse können sich gegenseitig "sehen"
	- -> Problem: Ein Prozess mit root-Privilegien, kann andere Prozesse beeinflussen
<br/>

- Wie kann ein Cloud-Provider Kunden vollen Zugriff auf ein System geben? 
	- Ohne dabei andere Sicherheitsrisiken für andere Kunden zu schaffen? 
<br/>

> Namespaces bilden die Grundlage für ==Container== (leichtgewichtige Virtualisierung)
- Namespaces abstrahieren die Sicht auf globale Ressourcen des BS
	- z.B. Filesysstem-Mount-Points, Netzressourcen, PIDs, etc.
<br/>

- Prozesse durch Namespaces vom Rest des BS isoliert
- Eine Gruppe von Prozessen hat die Illusion alleine auf dem Sys ausgeführt zu werden
- Linux-Namespaces sind ähnlich zu Solaris Zones und FreeBSD Jails
<br/>

- Namespaces werden als "==Wrapper==" für die zuvor globalen Ressourcen implementiert
- -> Jede Ressource ist eindeutig innerhalb eines Namespace-Containers
<br/>

> 6 unterschiedliche Typen von Namespaces von Linux unterstützt
- UTS: Unix Time Sharing (System)
	- Isolieren den Host- und Domainnamen
- IPC: Inter-Process Communication
	- Isolieren IPC Ressourcen (z.B. POSIX Message Queues)
- Network
	- Isolieren Netzressourcen
- PID: Process Identifier
	- Isolieren den PID-Raum
- Mount
	- Isolieren Mount-Points des Dateisys
- User 
	- Isolieren User- und Group-ID-Räume
<br/>

- Unterscheidung: ==hierarchische== und ==nicht-hierarchische== Namespaces

#### UTS-Namespaces
- Isolieren System-Identifier
- Prozesse in unterschiedlichen UTS-Namespaces können verschiedene eigene Host- und NIS Domainnamen, Architekturtypen, etc. haben
- siehe `{c} struct new_utsname`
- UTS wurde vom `{c} struct utsname` abgeleitet
	- struct hält die System-Identifier
- `{c} struct utsname` wird nicht mehr global, sondern pro Namespace verwaltet
<br/>

- Wichtig für SSL-Zertifikate, Logging etc

> Wenn du auf einem normalen Linux-System `{c}sethostname()` aufrufst, **änderst du den Hostnamen für das gesamte System**. Das würde alle anderen Prozesse betreffen – genau das soll bei Containern verhindert werden.

##### Beispiel

```c
#define STACK_SIZE (1024 * 1024)
#define hostname "safe(name)space"
#define domainname "safe.name.space.de"

static char stack[STACK_SIZE];

int main()
{
	/* Create child in new UTS namespace (-> CLONE_NEWUTS). */
	pid_t pid = clone(child_function, stack+STACK_SIZE,
		CLONE_NEWUTS|SIGCHLD, NULL);

	printf("Child PID: %ld\n", (long)pid);

	/* Wait for child to finish. */
	waitpid(pid, NULL, 0);
	
	exit(EXIT_SUCCESS);
}
```

- `{c}clone()` ist wie `{c}fork()`, aber mit mehr Kontrolle
- Mit `{c}CLONE_NEWUTS` erhält der Kindprozess eine **Kopie** der `{c}struct utsname` – aber keine Referenz auf die globale. Änderungen daran beeinflussen nur diesen Namespace.

```c
static int child_function(void *arg)
{
	sethostname(hostname, strlen(hostname));  // "safe(name)space"
	setdomainname(domainname, strlen(domainname));  // "safe.name.space.de"
	
	printf("NS: PID = %ld\n", (long)getpid());  // -> 1
	printf("NS: PPID = %ld\n", (long)getppid());  // -> 0
	
	/* Start a new shell. */
	 execlp("/bin/bash", "/bin/bash", (char *)NULL);
}
```
- Das Kind ändert **seinen eigenen** Hostnamen und Domainnamen – ohne den Host zu beeinflussen.
- Das Kind ersetzt sich selbst durch eine Bash-Shell. Diese Shell **erbt** den UTS-Namespace des Kindes.

> Beim Ausführen: 

**Vorher** (Host-System):

```bash
$ hostname
mein-host-server
```

**Nach dem Start des Programms** – in der neuen Shell:

```bash
$ hostname
safe(name)space
$ domainname
safe.name.space.de
```

**In einem anderen Terminal** (während die Shell noch läuft):

```bash
$ hostname
mein-host-server   # Unverändert!
```

#### PID-Namespaces
- Isolieren PID-Räume
- Prozesse in ==unterschiedlichen PID-Namespacess== können ==dieselbe PID== haben
- PIDs innerhalb eines PID-Namespace sind eindeutig
	- beginnen jeweils bei 1 und werden sequenziell hochgezählt
	- PID 1 ist dem ==init==-Prozess zugeordnet (-> hat im PID-Namespace auch keine PPID)
	- Jeder Prozess hat neben seiner PID im PID-Namespace eine PID im ==Parent-Namespace==
	- Bsp: Ein Prozess mit PID 1 im PID-Namespace trägt PID 31372 im Parent-Namespace
<br/>

- -> PID Namespaces erlauben Container auf andere Systeme zu migrieren, ohne dass sich dabei die PIDs der Prozesse ändern
<br/>

> init-Prozess im PID-Namespace
- der erste Prozess in einem PID-Namespace erhält die PID 1
- init-Prozess hat hier ähnliche Rolle wie der unter Linux
	- Zuständig für die Initialisierung des PID-Namespaces (z.B. Starten von weiteren Prozessen)
	- Wird Elternprozess für alle verwaisten Prozesse im PID-Namespace
<br/>

> Kommunikation durch ==Signale==
- Signal-Handler ermöglichen dem init-Prozess Kommunikation mit 
	- Prozessen in demselben PID-Namespace
	- Prozessen in einem verwandten, hierarchisch übergeordneten Namespace
	- dem BS-Kern
- init-Prozess im PID-Namespace soll **nicht** durch bestimmte Signale unterbrochen werden -> **ignorieren** z.B. SIGTTOU
- BS-Kern und Prozesse in hierarchisch übergeordneten Namespaces können durch die Signale SIGKILL und SIGSTOP den init-Prozess **zwingen**, sich zu beenden
<br/>

> PID-Namespaces formen ==weitere Hierarchieebenen==
- Es können ==verschachtelte== (nested) PID-Namespaces erstellt werden
- Prozesse können nur die Prozesse innerhalb desselben PID-Namespaces und der Kind-Namespaces der nächsten untergeordneten Ebene "sehen"


#### IPC- und Network-Namespaces
> IPC-Namespaces isolieren **IPC-Ressourcen**
- IPC-Objekte in System V und POSIX PIC werden nicht durch Pfade im Dateisys identifiziert
- Jeder IPC Namespace verwaltet
	- Eine Menge von System V IPC-Bezeichnern (IPC Identifiers)
	- Ein POSIX-Message-Queue-Filesystem: /dev/mqueue
	- Shared-Memory-Bereiche
<br/>

> Network-Namespaces isolieren **Netzressourcen**
- Network-NS erlauben Prozessen unterschiedliche Netzkonfigurationen im selbem Kern
- Bsp: Firewallkonfig, IP-Adressen (und Portnummern), Routen
- Physische NICs können nicht einem Network-NS zugeordnet werden
- Stattdessen müssen virtuelle Geräte verwendet werden

#### Mount-Namespaces
- Isolieren Mount-Points des Dateisys
- Mount-NS waren der erste Namespaces-Typ unter LInux
- Prozesse in unterschiedlichen Mount-NS haben **verschiedene Sichten auf das Dateisys**
- Jeder Mount-NS verwaltet eine eigene Liste von Mount-Points, die individuell durch mount und umount verwaltet wird
<br/>

> Shared Subtrees bieten 4 unterschiedliche Mount-Typen
- Shared Mount
	- Mount- und Unmount-Events werden unter Peer Groups propagiert
- Private Mount
	- Der Mount-Point empfängt und leitet keine Events zu Peer Groups weiter
- Slave Mount
	- Slave-Mount-Points empfangen Events von einer Master Peer Group
- Unbindable Mount
	- Ein Private Mount, der nicht gebunden (Bind Mount) werden kann
<br/>

> Peer Groups
- Menge von Mount-Points, die Mount- und Unmount-Events untereinander propagieren
- Shared Mount-Points werden beim Erzeugen eines Mount NS zu einer Peer Group hinzugefügt
<br/>

> Bind Mounts
- Ein Bind Mount wird verwender, um eine Datei/ein Verzeichnis in einer anderen Stelle des Filesys sichtbar zu machen
- Bind Mounts sind unabhängig vom Dateisys

#### User-Namespaces
- isolieren User- und Group-IDs
- Prozesse können innerhalb von User-Namespaces andere Privilegien als auf dem Host Sys besitzen
	- z.B. können unprivilegierte Prozesse root-Privelegien in User-NS erhalten
- Erhöhte Privilegien sind **nur innerhalb** des User NS gültig
	- Falls ein root-Proz eines User-NS auf Ressourcen zugreifen möchte, die das gesamte Sys beeinflussen, so prüft der BS-Kern, ob der Prozess entsprechende Privilegien außerhalb des NS besitzt
- Ähnlich zu PID-NS, können auch verschachtelte User-NS erstellt werden
	- Ein Proz kann unterschiedliche User- und Group-IDs auf jeder Ebene haben
	- Privilegierte Operationen, die von User-NS nicht unterstützt werden, können nicht ausgeführt werden
		- z.B. Kernel Module laden, Syszeit verändern, Sys rebooten, etc.
<br/>

- Aber: User Namespaces sind sehr komplex, erweitern den Angriffsvektor
	- Mehrer CVEs (common vulnerabilitiers and exposures) sind kurz nach der Veröffentlichung der User-Namespaces entstanden
	- -> In vielen Linux-Derivaten per Default deaktiviert


### Linux Control Groups (cgroups) 
> unterteilen Prozesse in hierarchische Gruppen
- Allozieren und verteilen Sysressourcen in Prozessgruppen
	- CPU-Zeit, CPU-Cores
	- Speicher
	- Festplatten- und Netz-Bandbreite
- Ressourcen durch unabhängige cgroup-Subsys repräsentiert
	- cgroup-Subsysteme (Resource Controller) steuern das Verhalten von Prozessgruppen
	- Sysressourcen können begrenzt werden und werden überwacht
	- Jedes Subsys verwaltet eine Hierarchie an Prozessgruppen
<br/>

> cgroup-Subsysteme
- blkio, cpu, cpuacct, cpuset, debug, devices, freezer, memory
- net_cls, net_prio, pids
- ...
<br/>

> Accounting
- Überwacht Seiten, die von den Prozessgruppen verwendet werden
	- Anonymous Pages: z.B. Stack und Heap
	- Active Pages: vor kurzem verwendete Pages
	- Inactive Pages: Kandidaten zum Auslagern
- Seiten können unter mehreren Gruppen unterteilt werden
	- In dem Fall werden die "Kosten" auf die Gruppen verteilt
<br/>

> Kontrolle
- Jede Prozessgruppe kann ==weiche== und ==harte Speichergrenzen== besitzen
- Weiche Grenzen sorgen dafür, dass Pages wieder zurückgegeben werden (kein Forcieren)
- Harte Grenzen triggern einen (ggf. angepassten) Out-of-Memory-Killer 
	- Alle Proz in der Gruppe werden eingefroren
	- Out-of-Memory-Killer kann Prozesse beenden oder Speichergrenzen anpassen
	- Wird die Grenze wieder unterschritten, können Prozesse fortgesetzt werden
- Speichergrenzen können z.B. für Kern- oder physischen Speicher gesetzt werden


### Secure Computing Mode (Seccomp)
> Mechanismus im Linux-Kern, um ==Syscalls für Prozesse einzuschränken==
- 3 Modi: Disables, Strict und Filter
- Modus in /proc/PID/status unter dem Eintrag Seccomp erkennbar
- Aktiviert durch die Syscalls `{c}prctl()` oder `{c}seccomp()`
- Eingesetzt u.a. in Docker, Chrome Browser, OpenSSH und Firefox OS
<br/>

> Strict Mode: SECCOMP_MODE_STRICT
- Schränkt den Prozess auf ==vier Syscalls== ein
	- `{c}read()`, `{c}write()`, `{c}exit()` und `{c}sigreturn()`
- Alle weiteren Syscalls triggern ein SIGKILL Signal
- Beachte: `{c} open()` nicht erlaubt
	- D.h. Dateien müssen **im voraus** geöffnet werden
<br/>

> Filter Mode: SECCOMP_MODE_FILTER
- Steuert, welche Syscalls für den Prozess erlaubt sind
- Filter basiert auf dem ==Berkeley Packet Filter (BPF)==
<br/>

> Berkeley Packet Filter (BPF)
- Implementiert als Virtuelle Machine bzw. Interpreter im Linux Kernel
	- Interpretiert eine sehr kleine ISA
	- Feste Instruktionsgröße
	- Filter werden in Form von kleinen BPF-Programmen realisiert
- Ursprünglich als Paketfilter für netzt u.a. in tcpdump eingesetzt
	- Effizientes Filtern von Netzpaketen im Kernel Space
	- Ungewollte Pakete werden früh (im Kernel) verworfen, also gar nicht erst an den User Space übergeben
	- Heute auch eingesetzt für: Container Security, System Tracing, Paketfiler, ...
<br/>

> BPF in Seccomp
- Jeder Syscall wird gegen installierte Filter getests
- Filtert nach ==Systemcall-Numern== und ==Argument-Werten==
- Installierte Filter können nicht deaktiviert werden

# System Virtual Machines

- ==Virtual Machine Monitor== (VMM) - ==Hypervisor== - bietet eine Laufzeitumgebung für BS
	- **Type 1 Hypervisor**: Direkt auf der Hardware ausgeführt
		- z.B. Xen, Hyper-V, VMware ESX
	- **Type 2 Hypervisor**: Verwendet Dienste des Host-BS
		- z.B. KVM, VirtualBox, VMware Workstation
	- ![[Pasted image 20260405175208.png|625]]
<br/>

> Der Hypervisor implementiert eine virtuelle ISA
- Virtuelle (Source) ISA besteht aus der User- und System-ISA
- Source ISA kann (muss aber nicht) der Target-ISA des Systems entsprechen
<br/>

> Aus der Sicht des Gast-BS wird das "System" durch die HW unterhalb der ISA-Schnittstelle charakterisiert

> VMM ==virtualisiert die gesamte physische Maschine==
- Schnittstelle ist die (virtualisierte) HW
	- Gast-BS arbeitet auf dieser
- Illusion, dass das Gast-BS vollständige Kontrolle über die HW hat
	- VMM behält natürlich die Kontrolle
- Gast-BS ist "Anwendung" aus Sicht des VMM
<br/>

> Implikationen
- Eine VM kann ein beliebiges BS ausführen
- Mehrere Instanzen des gleichen/verschiedener BS können gleichzeitig laufen
- Praktischer Nutzen siehe Linux-VM in VirtualBox

## Gründe für System-VMs

> Emulation
- BS können auf verschiedenen ISAs ausgeführt werden
- Debuggen von BS in Virtuellen Maschienen ist oft bequemer als auf der echten Hardware ("Bare-Metal")
- ISA-Design: eine ISA ist einfacher in SW zu implementieren als in HW
<br/>

> Isolation
- Verschiedene BS laufen isoliert in einer VM auf der gleichen HW
- Sandboxing von Gast-BS
- Ein kompromittiertes Gast-BS kann weder den Zustand weiterer VMs auf dem System noch den Hypervisor selbst manipulieren
<br/>

> Sicherheit
- Security-Mechanismen im BS können durch Malware umgangen oder ausgeschaltet werden
- Hypervisor ist leichter zu sichern (kleiner, höhere Privilegien)
- Sicherheitsmechanismen für Gast-BS können im Hypervisor plaziert werden
<br/>

> Ressourcennutzung
- HW-Ressourcen können besser (flexibler) ausgeschöpft werden

## Herausforderungen

> Der Hypervisor kontrollier und virtualisiert HW-Ressourcen des Systems
- CPU-Virtualisierung
- Speicher-Virtualisierung
- I/O-Virtualisierung
<br/>

> Ressourcen werden gemultiplext und unter allen VMs aufgeteilt
- Jede Gast-VM erhält virtuelle HW-Ressourcen
- Der Gast darn nicht in der Lage sein, auf andere als die ihm zugeteilten Ressourcen zuzugreifen
- Der Hypervisor vermittelt dem Gast-BS die Illusion, alleine auf dem Sys ausgeführt zu werden

## CPU-Virtualisierung 
> Paravirtualization
- Modifiziertes BS in VM; BS ist bewusst, dass es virtualisiert wird
- Dedizierte Geräte Treiber kommunizieren mit dem Hypervisor durch sog. ==Hypercalls==
- Kooperation zwischen dem BS in der Gast VM und dem Hypervisor
<br/>

> Binary Translation
- Unmodifiziertes BS kann ausgeführt werden
	- Ggf. auf verschiedenen Architekturen
- Hypervisor interpretiert/emuliert (Teile des) Binärcode(s) der Gast-VM in Software: ==Performance-Overhead==
	- Reduziert durch Caching, Beschränkung auf Code vom Gast-BS
	- Ggf. effizienter als Traps
- Hohe Komplexität im Hypervisor
<br/>

> Hardware-assisted Virtualization
- Hardware-unterstützte Virtualisierung von unmodifizierten BS
- HW Virtualization Extensions: Intel VT-x, AMD-v, ARM
- Ermöglichen das Implementieren von effizienten VMS
<br/>

> System und User Mode
- Man unterscheidet zwischen dem hochprivilegierten ==System Mode== und dem ==User Mode==
- Nicht zu verwechseln mit **Kernel Space** und **User Space** des BS
- Der Hypervisor wird im System Mode ausgeführt (z.B. Intel VMX root)
- Gast-BS wird im User Mode ausgeführt (z.B. Intel VMX non-root)
<br/>

> Bestimmte Instruktionen, die im User Mode ausgeführt werden, werden vom Hypervisor im System Mode abgefangen
- Hypervisor behält somit die Kontrolle über die VM
- ![[Pasted image 20260405201951.png|572]]
<br/>

### Instruktionsklassen
- Unterteilung der Instruktionen einer ISA in 3 Klassen (Popek & Goldberg)
<br/>

> Privileged Instructions
- Können nur im System-Mode ausgeführt werden
- Werden im Sys-Mode abgefangen, wenn sie im User-Mode ausgeführt werden: Trap
<br/>

> Sensitive Instructions
- Control-Sensitive Instructions
	- Modifizieren die Sys-Konfiguration (z.B. Page Tables)
- Behaviour-Sensitive Instructions
	- Verhalten sich anders, wenn sie im User-Mode ausgeführt werden
<br/>

> Innocuous Instructions
- Nicht priviligierte oder sensitive Instruktionen (alle anderen)
<br/>

> Für Virtualisierung müssen sensitive Instruktionen eine **strikte Untermenge** der privilegierten Instruktionen sein
- Gegenbeispiel bei x86: POPF (Flags-Register vom Stack in die CPU laden)
- Flags umfassen das Interrupt-Bit (enable/disable): wird im User-Mode nicht modifiziert


### Virtualisierungstechnologien

> Virtualisierungstechnologien (VT) in CPUs seit 2005
- (AMD Secure Virtual Machine, SVM) - IBMs seit dem 1960ern
- Gast-BS läuft in einem "Container" (!= Docker)
- Hypervisor definiert eine Bitmap mit nicht zugelassenen Instruktionen
- Jede dieser Instruktionen führt zu einem Trap
- ==Trap-and-Emulate== wird möglich
	- Hypervisor inspiziert die Instruktion und führt dann die entsprechenden Schritte (nach vorheriger Prüfung durch)
<br/>

> Und vorher? (VMware ist aus den 1990ern)
- Dynamisches Umschreiben von Instruktionen (on-the-fly)
- Ersetzen von privilegierten/sensitiven Instruktionen im Binärcode durch Traps
	- Normalerweise auf das Gast-BS beschränkt (keine Anwendungen)
	- Die meisten Instruktionen müssen nicht angepasst werden
- Prüfung der Operationen und dann Emulation
- Binary Translation

## Speicher-Virtualisierung
- Betriebssystem hat normalerweise uneingeschränkte Kontrolle über den physischen Speicher
	- Verwalten virtuellen Speicher für Prozesse
	- Seitentabellen zur Abbildung auf den physischen Speicher
<br/>

- Durch Virtualisierung konkurrieren plötzlich mehrere BS
	- Hypervisor ist die übergeordnete Instanz
<br/>

> Hypervisor ==teilt== den physischen Speicher zwischen VMs auf
- BS darf nicht auf beliebige Seiten im physischen Speicher zugreifen 
- BS soll nur den Speicher "sehen", den der Hypervisor zur Verfügung gestellt hat
- Hypervisor muss den Zugriff des BS auf Seitentabelle überwachen
<br/>

> Verschiedene Virtualisierungsansätze
- Shadow Page Tables (SPT)
- HW-unterstützte Second Level Address Translation (SLAT)

### Shadow Page Tables (SPT)
> Drei Abstraktionsebenen für Speicher
- Physischer Speicher im System (Host-Physische Adressen, HPA)
	- z.B. 32GB
- Abstraktion des physischen Speichers durch den Hypervisor
	- ==Gast-Physische Adressen==: Der Speicher, den ein Gast-BS sieht
		- z.B. 8GB kontinuierlicher Speicher
		- (nicht unbedingt kontinuierlich im physischen Speicher)
	- Virtueller Speicher: wie zuvor für Prozesse bescprochen
		- 4 GB Standardadressbereich
<br/>

> Das Gast-BS verwaltet eigene Seitentabellen (Gast-PT)
- Diese Gast-Seitentabellen werden ==**nicht**== von der MMU verwendet
- Hypervisor verwaltet eine ==Shadow Page Table pro Prozess==
	- SPT bildet virtuelle Adressen der Gast-VM auf physische Adressen ab
- Änderungen in der Gast-PT müssen mit SPT synchronisiert werden
	- Der Hypervisor gewährt der Gast VM Nur-Lese-Zugriff auf die Page Tables (read-only)
	- Jeder Schreibversuch wird vom Hypervisor abgefangen -> **Trap**! 
	- Änderungen werden sowohl in der Gast-PT als auch in der SPT übernommen
- -> Problem: Sehr komplexe Verwaltung in Software und hoher Overhead

### Second Level Address Translation 
> HWunterstützung in der CPU
- Zweite Stufe der Adressabbildung über SLAT-Tabellen
- Bildet Gast-Physische-Adressen (GPA) auf Host-Physische-Adressen (HPA) ab
- HV fänge Zugriffe auf Speicherbereiche ab, die nicht in den SLAT-Tabellen abgebildet sind

### Speicherverwaltung für VMs
- Wie viel Speicher soll eine VM bekommen? 
	- Gleiche Fragestellung wie für Prozesse
<br/>

> Statische Allokation
- Inflexibel, bindet Ressourcen
<br/>

> Dynamische Allokation
- Gleiches (optimistisches) Prinzip: Overcommitment
	- Annahme: nicht alle VMs benötigen gleichzeitig ihren gesamten Speicher
- Einfach: Speicher bei Bedarf zuweisen
- Schwierig: Speicher wieder wegnehmen, wenn nicht mehr benötigt
	- Paging? - HV weiß nicht, welche Seiten wichtig/unwichtig sind
- Lösung ==Ballooning== ("balloon driver")
	- SW-Komponente innerhalb der Gast-VM kommuniziert mit Hypervisor
	- Alloziert Seiten (Ballon bläht sich auf), wenn der HV Speicher braucht
	- Gibt Seiten frei (Ballon schrumpft), wenn der HV Speicher zur Verfügung hat
	- Gast-BS trifft die Entscheidungen über Ein-/Auslagerungen
<br/>

> Orthogonal: Seiten VM-übergreifend nutzen
- "Deduplication" - identische Seiten erkennen -> teilen + copy-on-write

## I/O-Virtualisierung 
### Ziele
- I/O-Geräte allen VMs/Gast-BS zur Verfügung stellen+
- Zugriff auf I/O-Geräte entweder durch SW oder HW gemultiplext
- Verhindern, dass Gast-VMs die Kontroller über I/O-Geräte übernehmen

### I/O-Virtualisierungstechniken
> Full Virtualization / Emulation
- Virtuelle Ethernet-Schnittstelle pro VM
	- Eigene Adressen, volle Kontrolle durch die VM
	- Ethernet-Switch im Hypervisor zum Verbinder aller VMs mit der Außenwelt
- Emulation von Festplatten, CD/DVD-Laufwerken in der VM
	- Abbildung auf Dateien im Host-BS
<br/>

> Paravirtualization 
- Verwendet eine ==Spllit-Driver-Architektur==
- Backend-Triber: Dedizierte privilegierte VM greift direkt auf Geräte zu 
- Frontent-Treiber: In Gast-VMs kommuniziert mit dem Backend-Treiber
- Vorteile
	- Geringer Overhead
	- Im Vergleich zu Full Virtualization leichter zu implementieren 
	- Adaption für Gast-BS: z.B. neue Geräte mittels bestehender Treiber anbieten
- Probleme
	- Benötigt Modifikationen/Treiber im Gast-BS
<br/>

> Device Domains
- Eine VM dient als Schnittstelle zu allen Geräten (Domain 0 bzw. "dom0" bei Xen)
- Andere VMs greifen über die VM zu 
- In Kombi mit Paravirt: semantische Kommandos (lese Block 1507) statt Sequenzen von Registerkommandos
<br/>

> Direct I/O
- HW-unterstütztes Multiplexing von I/O-Geräten
- Dedizierte I/O-MMU verwaltet Geräte-Speicher in Gast VMs
	- Bildet I/O-Adressen, Interrupts usw. ab
	- Isoliert DMA-Zugriffe
	- Beispiele: Intel VT-d, Intel VT-c, AMD IOMMU, ARM System MMU
	- Vorteile 
		- Gast-VMs greifen direkt auf die HW zu: Effizient
		- HW übernimt Teile des Multiplexings: Leicht zu implementieren
<br/>

> Single Root I/O Virtualization (SR-IOV)
- Direktes Zuweisen von HW zu einer VM skaliert nicht
- Hypervisor beim I/O-Zugriff umgehen
- Geräte unterstützen Virtualisierung direkt
	- Erscheinen als mehrere Instanzen
	- Jede steht einem Gast-BS exklusiv zur Verfügung
- Physical Functions (PF) für den HV (voller Zugriff)
- Virtual Functions (VF) für die Gast-VMs (einfeschränkter Zugriff)



# Process Virtual Machines
> Ansatz
- Runtime implementiert eine virtuelle Laufzeitumgebung für User-Space-Prozesse
- Die Runtime implementiert das ABI-Interface bestehend aus dem Systemcall-Interface und der User-ISA (Plattform)
- Aus der Applikationssicht wird das "System" durch das ABI-Interface charakterisiert
<br/>

> Gründe für Process VMs
- Emulation: Programme können auf verschienen ISAs ausgeführt werden
- Plattform-Unabhängigkeit: Programme können auf verschiedenen BS ausgeführt werden
- Performance-Optimierung: Binary Optimizer optimieren den Code währen der Laufzeit
<br/>

> Virtuelle Umgebungen für einen (oder mehrere) Prozesse
- Anwendungen auf verschiednen Plattformen ausführen
- Beispiel: Intel IA-32 Execution Layer (IA-32 EL) erlaubt es, x86-32-Bit Anwendungen auf 64-bit-Itanium auszuführen
<br/>

> High-level-Language (HLL) VMs
- HLL VMs definieren eine Plattform-unabhängige virtuelle ISA
- Verhindert Abhängigkeiten zum BS und zur HW-Architektur
- Beispiel: Java Virtual Machine (JVM) interpretiert Java-Bytecode
